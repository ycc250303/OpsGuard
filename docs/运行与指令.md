# OpsGuard 运行与指令说明

Linux 主机 + Docker Compose 运维诊断 Agent。用户以自然语言提问，Agent 通过 ReAct 调用只读工具，经 **Collector → Analyzer → Advisor** 三阶段流水线输出 Markdown 诊断报告。

> 设计细节见 [`project-plan.md`](project-plan.md)；Agent 协作约定见根目录 [`AGENTS.md`](../AGENTS.md)。

---

## 1. 环境要求

| 项 | 要求 |
| --- | --- |
| 开发机 | .NET **10** SDK（`dotnet --version`） |
| 运行主机 | **Linux**，且与 Docker 同机（Agent 读 `/proc`、执行 `docker`） |
| Docker | 运行用户须在 `docker` 组，可执行 `docker inspect` / `docker logs` |
| LLM | 阿里云百炼 API Key，环境变量 `DASHSCOPE_API_KEY` |
| 网络 | 运行主机可访问 `https://dashscope.aliyuncs.com` |

**重要：** 在 macOS/Windows 本地可编译、跑单元测试与 LLM 连通性测试，但 **Host 指标与 Compose 诊断须在 Linux 目标机上执行**（当前生产机：`111.229.81.45`）。

## 2. 首次配置

### 2.1 克隆与构建

```bash
git clone <repo-url> OpsGuard
cd OpsGuard
dotnet restore
dotnet build
dotnet test
```

### 2.2 API Key（`.env`）

```bash
cp .env.example .env
# 编辑 .env，填入：
# DASHSCOPE_API_KEY=sk-xxx
```

- `.env` 已在 `.gitignore` 中，**勿提交 Git**
- 启动时自动加载：仓库根目录 + 程序所在目录（部署目录）
- 若 Shell 已 export 同名变量，**以环境变量为准**（不被 `.env` 覆盖）

### 2.3 Compose 拓扑 JSON

| 文件 | 用途 |
| --- | --- |
| `docs/compose-topology.sample.json` | 仓库示例，可提交 |
| `docs/compose-topology.production.json` | 生产配置，**不进 Git** |

**首次填写容器名：**

```bash
# 在目标 Linux 服务器上
docker ps --format '{{.Names}}'
```

将输出填入 JSON 的 `ContainerName`，`Id` 为工具参数 `serviceId`（如 `backend`）。有 HTTP 入口的服务填写 `HealthUrl`（如 `http://127.0.0.1:8080/actuator/health`）。

字段说明见 [`project-plan.md` §2.5](project-plan.md#25-compose-拓扑配置)。

### 2.4 应用配置（`appsettings.json`）

| 配置节 | 键 | 默认 | 说明 |
| --- | --- | --- | --- |
| `Llm` | `ModelId` | `qwen-plus` | 百炼模型 ID |
| `Llm` | `Endpoint` | 百炼 OpenAI 兼容地址 | 一般无需改 |
| `Agent` | `MaxLogTailLines` | `200` | 日志行数上限 |
| `Agent` | `OrchestrationTimeoutMinutes` | `10` | 单次诊断超时（分钟） |
| `Agent` | `ConversationHistoryTurns` | `6` | 多轮对话带入的历史轮数 |

## 3. 命令行参数

```
dotnet run --project src/OpsGuard.App -- [选项]
```

| 参数 | 说明 |
| --- | --- |
| `--topology <path>` | 拓扑 JSON 路径；省略时使用 `docs/compose-topology.sample.json` |
| `--query "<问题>"` | 单次非交互诊断，执行后退出 |
| `--smoke` | 冒烟测试：API Key、Host Plugin、LLM、Tool Calling |
| （无参数） | 进入交互模式，提示符 `ops>` |
| `exit` / `quit` | 交互模式下退出 |

**部署后等价命令：**

```bash
/root/.dotnet/dotnet OpsGuard.App.dll --topology /path/to/topology.json [选项]
```

## 4. Web UI（Blazor Server）

除命令行外，可使用浏览器聊天界面（`src/OpsGuard.Web`）。

### 4.1 本地启动

```bash
# 确保项目根目录 .env 已配置 DASHSCOPE_API_KEY
dotnet run --project src/OpsGuard.Web
```

浏览器打开 **http://localhost:5299**。支持多轮对话、Markdown 报告渲染、清空会话；Enter 发送，Shift+Enter 换行。

指定拓扑（任选其一）：

```bash
# 环境变量
export OPSGUARD_TOPOLOGY=docs/compose-topology.production.json
dotnet run --project src/OpsGuard.Web

# 或启动参数（与 Console 相同）
dotnet run --project src/OpsGuard.Web -- --topology docs/compose-topology.production.json
```

### 4.2 服务器部署 Web（端口 5229）

Web 与 Console **同样检测运行所在机器**。生产诊断请在 `111.229.81.45` 上运行 Web 进程。

一键部署（推荐）：

```bash
./deploy/deploy-web.sh
```

脚本会：Release 发布 → 同步至 `/opt/opsguard-web/` → 复制 `.env` → 通过 Docker 镜像提取 ASP.NET Core 10 运行时（若缺失）→ 注册并启动 `opsguard-web` systemd 服务。

手动步骤：

```bash
dotnet publish src/OpsGuard.Web/OpsGuard.Web.csproj -c Release -o /tmp/opsguard-web
rsync -av /tmp/opsguard-web/ root@111.229.81.45:/opt/opsguard-web/ \
  -e "ssh -i ~/.ssh/github_actions_deploy -o BatchMode=yes"
scp -i ~/.ssh/github_actions_deploy .env root@111.229.81.45:/opt/opsguard-web/.env
scp -i ~/.ssh/github_actions_deploy deploy/opsguard-web.service root@111.229.81.45:/etc/systemd/system/
ssh -i ~/.ssh/github_actions_deploy root@111.229.81.45 \
  'systemctl daemon-reload && systemctl enable --now opsguard-web'
```

访问：**http://111.229.81.45:5229/**（生产拓扑：`/opt/opsguard/compose-topology.production.json`）

> **安全提示**：Web UI 无登录鉴权，勿对公网裸露。建议 SSH 隧道：`ssh -L 5229:127.0.0.1:5229 -i ~/.ssh/github_actions_deploy root@111.229.81.45`，浏览器打开 http://localhost:5229

Agent 与 Docker **必须同机**。当前部署目录：`/opt/opsguard/`。

### 6.1 发布与同步

在开发机执行：

## 8. GitHub Actions CD

`main` 分支 push 后自动部署 **App（/opt/opsguard）** 与 **Web（/opt/opsguard-web:5229）**。

### 8.1 仓库 Secrets（Settings → Secrets and variables → Actions）

| Secret | 必填 | 说明 |
| --- | --- | --- |
| `OPSGUARD_SSH_HOST` | 是 | 服务器 IP，如 `111.229.81.45` |
| `OPSGUARD_SSH_PRIVATE_KEY` | 是 | 部署用 SSH 私钥全文（对应服务器 `authorized_keys`） |
| `DASHSCOPE_API_KEY` | 否 | 若配置，每次部署写入服务器 `.env`；不配置则保留服务器已有 `.env` |

可选：在 GitHub **Environments → production** 中配置上述 Secrets 并启用保护规则。

### 8.2 工作流说明

- 文件：`.github/workflows/cd.yml`
- 并发组 `opsguard-server-deploy`，避免并行部署冲突
- 流程：单元测试 → rsync 发布 App/Web → 重启 `opsguard-web` → 远程 `--smoke` → Web HTTP 健康检查
- **不会**从仓库同步 `compose-topology.production.json`（本地/服务器维护，已在 `.gitignore`）

### 8.3 本地手动部署

```bash
./deploy/deploy-all.sh          # App + Web
./deploy/deploy-app.sh          # 仅 CLI/App
./deploy/deploy-web.sh          # 仅 Web UI
```

## 7. 诊断工具（Plugin）

Agent 仅能通过以下 5 个只读工具采集事实（`serviceId` 必须来自拓扑 JSON）：

| 工具 | 作用 |
| --- | --- |
| `GetHostMetrics` | CPU / 内存 / 磁盘 / 负载 |
| `GetComposeTopology` | 返回 JSON 中全部可检查服务 |
| `GetComposeServiceStatus` | 容器 running/exited、重启次数 |
| `QueryComposeServiceLogs` | `docker logs --tail N`（N ≤ 200） |
| `CheckHttpEndpoint` | 对配置了 `HealthUrl` 的服务 HTTP GET |

Console 会打印 `[Tool] Plugin.Function` 调用链；Multi-Agent 阶段标题为 `=== Collector ===` / `=== Analyzer ===` / `=== Advisor ===`。

## 9. 示例场景

| 用户输入 | 典型 Tool 行为 |
| --- | --- |
| 检查本机和所有 compose 服务 | `GetHostMetrics` + `GetComposeTopology` + 各服务 `Status` |
| backend 容器挂了，查日志 | `GetComposeServiceStatus` → `QueryComposeServiceLogs` |
| web-gateway 健康检查失败 | `CheckHttpEndpoint` → 必要时查 logs |
| 磁盘呢？（追问） | 结合对话历史，再次 `GetHostMetrics` |

## 11. 常见问题

| 现象 | 处理 |
| --- | --- |
| `DASHSCOPE_API_KEY environment variable is required` | 检查 `.env` 或 `export DASHSCOPE_API_KEY` |
| `Topology file not found` | 确认 `--topology` 路径；生产文件是否已复制到服务器 |
| `Host metrics are only supported on Linux` | 在 Linux 目标机运行，勿在 macOS 本地做 Host 诊断 |
| `Unknown serviceId` | 先 `GetComposeTopology`；检查 JSON 中 `Id` 是否与提问一致 |
| `docker inspect/logs failed` | 确认用户在 `docker` 组；`ContainerName` 与 `docker ps` 一致 |
| 诊断超时 | 缩小问题范围；可调大 `OrchestrationTimeoutMinutes` |
| Plugin 重名崩溃 | 已修复：每个 Plugin 使用独立名称（如 `ComposeStatus`） |

## 11. 安全说明

- 默认**只读**：不 restart / rm 容器，不执行任意 Shell
- LLM 只能传 `serviceId`，不能传任意容器名或路径
- API Key 仅通过 `.env` 或环境变量注入，勿写入 `appsettings.json` 并提交
- 服务器 Agent 文件操作：允许新增/覆盖，**禁止**删除（见 `AGENTS.md`）
